## Product Alignment Report

### Product Purpose

**Issue:** Product identity boundary is unclear
**Severity:** Critical
**Evidence:** README.md:3 — "Prati is the UI layer for the Astra ecosystem." However, architecture docs repeatedly refer to the project as "Astra" itself (e.g., `docs/raw/architecture/core/theming.md:1`: "Astra provides a MUI-based theming system", `docs/raw/architecture/invariants/stateless-ui.md:5`: "Astra is a presentation-oriented UI component library"). The design system doc is titled "Astra Design System" (`docs/raw/design-system/design.md`). Error messages in source code also say "Astra" (`src/common/localization/LanguageContext.ts:23`: "useLanguage must be used within a AstraLanguageProvider").
**Recommendation:** Decide whether the repo is "Prati" or "Astra" and normalize all docs and source code to one name. If Prati is a subset of Astra, the relationship must be precisely documented.

---

**Issue:** "What the product is not" is missing
**Severity:** Minor
**Evidence:** README does not state boundaries — what does Prati not do? No explicit non-responsibilities section.
**Recommendation:** Add a "What Prati is not" section (e.g., not a routing library, not a state management solution, not a backend service).

---

**Issue:** Positioning as "UI layer for Astra" contradicts feature scope
**Severity:** Major
**Evidence:** README.md:131-136 frames Prati as the UI layer paired with Astra. But Prati includes Proto Runtime (`src/proto-runtime/`), a Handlebars template renderer (`src/services/templateRenderer.ts`), and server-side templates (`src/templates/`) — features that go beyond a pure "UI layer" and suggest a broader scope.
**Recommendation:** Either expand the positioning to encompass these features, or clarify they are supplementary.

## Architecture Validation Report

### System Boundaries

**Issue:** README's project structure omits major subsystems
**Severity:** Major
**Evidence:** README.md:151-159 shows a project structure with only `src/common/components/`, `src/common/localization/`, `src/common/theme/`, `src/theme/`, and `src/lib.ts`. Missing directories: `src/proto-runtime/`, `src/services/`, `src/templates/`, `src/types/` — all of which exist and are exported from `src/lib.ts`.
**Recommendation:** Update the project structure to reflect all source directories.

### Provider Hierarchy

**Issue:** README does not describe provider nesting requirements
**Severity:** Minor
**Evidence:** Architecture runtime maps (`docs/raw/architecture/runtime-maps/provider-hierarchy.md`) define a precise provider hierarchy: ThemeProvider > LanguageProvider > AuthProvider > Router > Page. README shows theming and localization in separate sections with no mention of ordering.
**Recommendation:** Add a provider hierarchy diagram or note about nesting order.

---

### Architectural Drift

**Issue:** Architecture docs reference "Astra" while README says "Prati" — identity drift
**Severity:** Major
**Evidence:** All 10 architecture docs under `docs/raw/architecture/` reference "Astra" as the product name. The README claims Prati is the UI layer of Astra. This creates ambiguity about whether Prati *is* Astra's UI layer, or whether these docs are for a different product.
**Recommendation:** Audit all architecture docs and replace "Astra" with "Prati" where appropriate, or clearly delineate the boundary.

---

**Issue:** Design system doc references non-existent directory paths
**Severity:** Major
**Evidence:** `docs/raw/design-system/design.md:1457-1462` references paths like `docs/raw/design/brand/`, `docs/raw/design/references/`, `docs/raw/design/rules/` that do not exist. The actual paths are under `docs/raw/design-system/`.
